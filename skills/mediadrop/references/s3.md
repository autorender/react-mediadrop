# `@mediadrop/s3`

Two `UploadTransport`s, no AWS SDK, no signing in this package — **your
backend signs URLs; this package only ever talks to those URLs** with
`XMLHttpRequest`. If a task implies the browser needs AWS credentials,
that's a sign the backend contract below is missing, not something to
work around by importing the AWS SDK client-side.

See [upload.md](upload.md) for the shared queue/concurrency/retry/cancel
contract this plugs into.

## `s3Upload({ getUploadUrl })` — simple, single request

For files small enough for one request. Presigned PUT (raw body) or
presigned POST (S3's policy-based form upload) — same shape as
`@mediadrop/xhr-upload`, just S3's two presigned-request styles:

```ts
import { s3Upload } from "@mediadrop/s3";

const transport = s3Upload({
	getUploadUrl: async ({ file }) => {
		const res = await fetch("/api/s3/presign", {
			method: "POST",
			body: JSON.stringify({ filename: file.name, contentType: file.type }),
		});
		return res.json(); // { url, method?, headers?, fields?, key?, bucket? }
	},
});
```

No retry here, or anywhere in this package — the queue's `retries` owns
that (see [upload.md](upload.md)).

## `s3MultipartUpload({ ... })` — large files, resumable metadata

Splits a file into parts (enforcing S3's real constraints: parts **≥ 5
MiB except the last**, **≤ 10,000 parts total** — `s3MultipartUpload`
adjusts your requested `partSize` up as needed to satisfy both), uploads
them with bounded concurrency, aggregates progress across parts without
double-counting, retries a failed part via `@mediadrop/core`'s shared
`withRetry` (never a second retry loop), and — with `sessionStore` — can
skip already-uploaded parts on a subsequent attempt, including after a
page reload if the same file is reselected:

```ts
import { s3MultipartUpload } from "@mediadrop/s3";
import { browserUploadSessionStore } from "@mediadrop/core";

const transport = s3MultipartUpload({
	createMultipartUpload: async ({ file }) => fetchFromYourBackend(file),
	getPartUploadUrl: async ({ key, uploadId, partNumber }) => fetchFromYourBackend(...),
	completeMultipartUpload: async ({ key, uploadId, parts }) => fetchFromYourBackend(...),
	abortMultipartUpload: async ({ key, uploadId }) => fetchFromYourBackend(...),
	sessionStore: browserUploadSessionStore(),
});
```

### Your backend's contract

Never wire `s3MultipartUpload` unless the backend actually implements all
four callbacks — don't stub one out and call the integration done:

| Callback | Typical backend call |
|---|---|
| `createMultipartUpload` | S3 `CreateMultipartUpload` → `{ uploadId, key }` |
| `getPartUploadUrl` | Sign one part's PUT URL → `{ url, headers? }` |
| `completeMultipartUpload` | S3 `CompleteMultipartUpload` → `{ key?, location? }` |
| `abortMultipartUpload` (optional) | S3 `AbortMultipartUpload` |

**Your bucket's CORS config must expose the `ETag` response header**
(`ExposeHeaders: ["ETag"]`) — without it, the browser can't read each
part's ETag from the PUT response, and `s3MultipartUpload` rejects with a
message saying exactly that. This is the single most common integration
failure for this transport — check CORS first if parts fail with an ETag
error.

## Resumability — the caveat to always attach

**What actually resumes:** with `sessionStore` set, re-uploading the
exact same file (matched by `fingerprint` — metadata-based, not file
contents, by default) — including after a page reload — skips
already-uploaded parts.

**What does not resume:** mediadrop cannot persist the file's bytes. If
the user doesn't reselect that exact file, there is nothing to resume —
they start over. There is also no pause: canceling discards the resume
session, so resuming only helps after an *unplanned* interruption (a
reload, a closed tab), not a deliberate cancel. Never say "fully
resumable" without this caveat.

Without `listUploadedParts`, a resumed upload trusts its local session
metadata as-is; pass it (an S3 `ListParts` call) to reconcile against
what S3 actually has first.
